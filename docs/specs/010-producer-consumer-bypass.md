# 010 Producer-Consumer Architecture with DataDome Bypass

## Goal

Move scheduled monitor execution from an in-process scheduler/executor to a Redis-backed producer-consumer flow, and make every public Vinted catalog request use the `curl_cffi` browser impersonation stack with a prepared persistent Vinted session bound to a sticky residential proxy identity.

## Scope

- `SchedulerRunner` is a producer: it evaluates active monitors, windows and jitter, then enqueues `MonitorTask` payloads in Redis.
- `TaskConsumer` workers are consumers: they block on the Redis task queue and call `execute_monitor_run()`; the run factory resolves the ready Vinted session for the selected proxy.
- Manual runs stay synchronous from the API, but use the same egress selection and `CurlCffiVintedCatalogProvider` stack as scheduled runs.
- Existing business logic remains unchanged: Redis seen cache, deduplication, filters, item persistence, opportunities, run events and monitor sessions.

## Interfaces

- Redis task queue: the producer atomically writes `LPUSH vinted:task_queue`, one pending marker per monitor and a payload-identity reverse marker. Each consumer reserves FIFO work with blocking `BLMOVE` into its own `vinted:task_queue:processing:{consumer_id}` list; terminal outcomes use exact-payload ACK, unexpected failures requeue atomically, and malformed payloads move to `:dead-letter`.
- HTTP provider: serial mode uses one `CurlCffiVintedCatalogProvider`/`curl_cffi.requests.Session` per run, loaded from or used to create a monitor-owned encrypted Vinted session; catalog document bootstrap, DataDome collector attempt, catalog API and serial details share that cookie jar/sticky. Explicit canary/parallel detail mode may clone the prepared context into at most two same-sticky lane providers and reconciles the final context as defined below. The egress diagnostic uses the same sticky proxy/impersonation but an isolated cookie-free HTTP session so Vinted cookies/tokens are never sent to the IP echo service.
- Browser profiles: coherent `impersonate`, `User-Agent`, `sec-ch-ua*`, ordered lowercase bootstrap headers and ordered lowercase API headers. The default runtime profile is Chrome 146, matching the latest valid Chrome 146 catalog HAR and the highest Chrome target supported by the installed `curl_cffi` build; older profiles remain available only as explicit development diagnostics.
- Pre-integration fingerprint gate: `scripts/verify_impersonation.py` uses the reusable ephemeral HTTP client with `impersonate="chrome120"` against public echo services before worker integration.
- Proxy sticky sessions: `PROXY_STICKY_USERNAME_TEMPLATE` defaults to `{username}-session-{session_id}`. The `session_id` comes from the persisted monitor-owned Vinted session, not a new UUID per run while that session is reusable.
- DataDome collector: automatic monitor session preparation extracts the observed `DATADOME_CLIENT_SIDE_KEY` from the catalog document when present, otherwise may download `static-assets.vinted.com/datadome/.../tags.js` for diagnostics, then calls the DataDome JS endpoint in the same `curl_cffi` session, sticky proxy identity, browser profile, locale, viewport and catalog document context. It attempts only the observed `ch` then `le` collector sequence with HAR-shaped lowercase headers. Missing `datadome` is blocking: the catalog API probe may still run for diagnostics, but the prepared session is saved as `incomplete` and cannot be reused.
- Catalog API probe: monitor session preparation calls `/api/v2/catalog/items` once after the catalog document bootstrap. The probe returns only redacted diagnostics, records explicit `catalog_api_probe_*` run events, and may mark a monitor-owned Vinted session usable only when the base context is complete and the API returns JSON. It never persists items, creates opportunities, or changes monitor metrics.
- Explicit monitor session preparation: `POST /api/monitors/{id}/vinted-session/prepare` runs only the monitor-owned Vinted session bootstrap, DataDome collector attempt and catalog API probe. It creates a `session_prepare` audit run and logs, but does not check Redis, does not seed the initial snapshot, does not persist catalog items, does not create opportunities and is excluded from monitor metrics and charts.
- Item detail runtime: every due retry and Redis-new candidate fetches `/items/{id}-{slug}?referrer=catalog` through the same prepared session. JSON-LD is fallback data and Next/React Flight records are discovered by structure plus matching item id. Business runs do not use `/api/v2/items/{id}/details`, which both supplied HARs confirm is unnecessary and Cloudflare-challenged.
- Item detail probe: `POST /api/monitors/{id}/items/detail-probe` accepts an item id or URL, performs the same public document fetch/parser used by business runs, and returns a safe field/source/completeness summary. It remains an audit run with no Redis seen changes, item persistence, opportunities, or monitor metrics.
- Catalog URL filter compatibility is enforced before any traffic-producing action starts a run. Manual run, baseline recalibration, explicit Vinted session preparation and item detail probe all reject unsupported catalog URL filters without creating a `runs` row.
- Catalog API runtime classification separates DataDome challenge, Cloudflare challenge, anonymous-session rejection, plain rate limit and generic request error. A catalog `429` is rate limit unless response headers, body or an error-response cookie carry challenge signatures. `Retry-After` is recorded only as safe diagnostics; it never authorizes a wait, refresh or second request.
- `WORKER_MAX_RETRY_ATTEMPTS` applies only to unexpected worker failures. A classified challenge, session rejection or catalog `429` is terminal on its first response and cannot enter worker escalation.
- Proxy-related task data includes only `proxy_profile_id` plus the opaque effective-identity token selected with it; the payload also carries safe task/source/trigger identity and scheduling fields. Its format is `v1:<positive monotonic generation>:<HMAC-SHA256 hex>`. The keyed digest covers canonical scheme, host, port, username, decrypted password, stored country-derived preset and `PROXY_STICKY_USERNAME_TEMPLATE`; the counter prevents ABA reuse across every change observed by an API edit or runtime reconciliation, even if values later return to their prior state. Full proxy transport, usernames, passwords and cookies are never written to Redis or run metadata, and the token does not reveal its canonical values or permit offline password verification without the application key.
- Runtime metadata: runs include `task_id`, `consumer_id`, `browser_profile`, `vinted_session_id`, `vinted_session_request_count`, `vinted_session_max_requests`, `proxy_session_id_prefix` and `attempt`; full proxy credentials and raw cookies/tokens are never returned or logged.
- Deployment safety gate: public Vinted catalog traffic is not sent directly from the host unless `VINTED_DIRECT_CATALOG_ENABLED=true`; the default is false.
- Geo context: Vinted ES catalog runs require an ES proxy profile by default. Proxy profiles declare country only; locale, `Accept-Language`, viewport and Vinted `x-screen` are resolved from internal country/domain presets and stored as read-only diagnostics. The ES preset keeps `locale=es-ES` and `x-screen=catalog`, but uses the observed Chrome 146 HAR `Accept-Language` value `en-GB,en;q=0.9`.

## Public anonymous session lifecycle

The maintained end-to-end map is in `docs/architecture.md`. This spec owns the product contract:

- A prepared session is anonymous public catalog context, never an authenticated Vinted account. `monitor_sessions` owns a launch/run interval; `vinted_sessions` owns a reusable HTTP identity. They are different state machines.
- PostgreSQL is authoritative for the durable lifecycle. Redis receives safe task/cache/retry data, including normalized public candidates, but never cookies, tokens, encrypted context or a cookie jar. Serial detail mode uses one active in-memory `curl_cffi` provider/jar. Explicit canary/parallel mode may clone the same prepared context and sticky into at most two lane providers, then reconciles the last successful context; canary additionally validates the resulting catalog session.
- Effective readiness requires durable `ready`, the same monitor, the same effective proxy/sticky identity, browser profile, country/locale/`Accept-Language`/viewport/`x-screen`, unexpired TTL, remaining use budget, decryptable payload and complete required context. A raw `status=ready` response is not sufficient proof.
- Selection and every durable context write are serialized with the monitor row. Acquiring a prepared session must consume one durable logical use before business traffic; this counter is not the number of HTTP requests made by a run.
- Automatic preparation creates a new sticky identity, uses an isolated cookie-free egress diagnostic, then one shared Vinted jar for catalog bootstrap, collector and probe. Only accepted JSON plus complete context becomes `ready`; every other persisted attempt is `incomplete` and is not reusable.
- Explicit preparation is an audit command. Validation failures return an HTTP error without a run. Once its audit run exists, an operational failure returns `201` with `Run.status=failed`; success returns the same shape with `Run.status=success`. It never implies baseline readiness or business execution.
- Reuse reconstructs the exact accepted transport identity, loads the encrypted context into memory and keeps that jar/sticky for catalog and detail requests. Ordinary response-cookie rotation may update the same durable row without resetting its use counter or changing sticky identity; a failure response never triggers refresh.
- The first Cloudflare or DataDome challenge from catalog or detail, anonymous-session rejection, contradictory context or catalog `429` is terminal for the task. It creates one failed run, invalidates any acquired session by replacing its encrypted payload with `{}`, and the consumer ACKs the exact reservation without another provider call, wait, refresh, escalation or requeue. A later normal scheduler task or explicit user launch is a new attempt; claimed detail candidates remain durable for that future attempt rather than being lost.
- Expiry, exhaustion, incomplete preparation, invalidation and archive are distinct reasons. They must be visible without exposing secrets, and retention/purge must be explicit rather than inferred from the `status` label.
- The PWA read model must identify the session the runtime can actually select for a monitor. A latest-created row per proxy is diagnostic history, not the canonical current session.

### Effective proxy identity and pre-provider fence

- `proxy_profiles.identity_generation` is a monotonic counter and `proxy_profiles.identity_fingerprint` is the current keyed digest. `vinted_sessions.proxy_identity_generation` stores their combined opaque token and binds each encrypted context to the exact effective proxy identity used to prepare it. A change to scheme, host, port, username, password, country-derived locale/`Accept-Language`/viewport/`x-screen`, or sticky username template advances the counter and changes the binding. Profile name, kind, capacity, telemetry and operational eligibility do not alter that transport identity.
- Updating an identity-bearing profile field takes an exclusive transaction-scoped advisory lock, then a `FOR NO KEY UPDATE` profile lock. It advances the generation and locks affected monitors and session rows in stable order before invalidating every associated prepared-session row in the same commit. The first selection after a process restart with a changed sticky template performs the same exclusive reconciliation for generations created with the old template; an invalid template fails settings validation and cannot become an ordinary penalized proxy error. Invalidating replaces encrypted context with `{}`; no credential or canonical identity is copied to Redis, events, errors or API output.
- Every traffic-producing command captures `proxy_profile_id` plus identity token at egress selection. Before writing its first run event, runtime takes the shared transaction-scoped advisory fence and revalidates existence, active state, cooldown, target country, canonical country preset and the captured token; the provider path remains under that ownership and revalidates the current profile before construction. Shared locks allow concurrent admitted runs for the same profile and remain held through the last provider use and its first durable post-traffic PostgreSQL commit. A business run may then release the fence while it finishes the Redis-to-terminal `finalizing` transition because no provider call remains. A concurrent profile edit therefore either wins before the fence and makes captured work fail without provider construction, or waits until the already-authorized traffic is durably closed.
- An absent, inactive, cooling-down, wrong-country, corrupt-preset or generation-mismatched profile is fail-stop. Runtime neither prepares a session nor silently selects another profile. A fresh command after a legitimate identity change may select the new generation, invalidate old contexts and prepare only that new identity.
- Manual launch, baseline recalibration, explicit preparation, detail probe and Redis consumer work use this same fence. Direct mode has no proxy generation. Response/challenge policy, durable refresh writes and the canonical PWA eligibility read model remain owned by 14.12.3, 14.12.4 and 14.12.5 respectively.

## Acceptance Criteria

- No runtime code imports or depends on `httpx`; provider tests use an injected fake `curl_cffi` session instead of `httpx.MockTransport`.
- Scheduler enqueues due monitors and does not execute HTTP or run business logic directly.
- A monitor has at most one queued or in-flight scheduled task. A later scheduler tick coalesces while its pending marker exists instead of growing duplicate backlog.
- Scheduler capacity includes valid ready/reserved tasks from previous ticks as well as SQL `running` rows, without double-counting the same `task_id`, so global and per-proxy limits cannot be exceeded by queue backlog.
- A worker process recovers every reserved-but-unacknowledged task before starting producer or consumer threads. A crash before run creation is redelivered; a terminal run with the same indexed `task_id` is acknowledged without new Vinted traffic; a `finalizing` run is reconciled; and an orphan `running` run is closed before redelivery.
- Queue ACK/requeue/dead-letter transitions are idempotent after an ambiguous Redis response. A consumer retries maintenance transitions with backoff, recovers only its own processing list after an ambiguous reservation, and never creates a second ready copy when another consumer already reserved the task. Invalid payloads use the reverse marker to release their monitor even when their JSON identity cannot be parsed, and remain in dead-letter for safe diagnosis.
- Queue transport uses a binary Redis client with a socket timeout longer than blocking reserve. Non-UTF-8 corruption survives startup recovery as exact bytes and is then dead-lettered instead of crashing or poisoning the consumer loop.
- Runs resolve a ready Vinted session for the selected monitor and proxy. If none exists, is expired, exhausted, invalid, or belongs to another monitor, the run automatically prepares a new sticky Vinted session before business processing.
- Ready-session selection requires the current effective proxy generation. A stale generation is invalidated and purged under proxy/monitor ownership before any replacement is considered; an ineligible captured profile stops without constructing or calling a provider.
- Real API/PostgreSQL/Redis race verification changes every identity input plus active state, cooldown and target eligibility between selection and the provider barrier for manual and queued work. A local provider trap/call counter must stay at zero for stale work, while a fresh post-change command may reach only the new safe generation. Tests restore settings/profile telemetry and delete their sessions, runs, events and queue keys without Vinted or proxy traffic.
- The PWA monitor detail exposes a `Preparar sesion` action while the monitor is stopped. It is blocked by unsaved monitor changes or unsupported URL filters, uses the same proxy/session stack as launch and recalibration, and leaves the result visible in accumulated logs.
- One classified Cloudflare/DataDome challenge, anonymous-session rejection or catalog `429` records safe diagnostics, returns a failed run and invalidates/purges the acquired Vinted session. Provider tests prove one request and zero failure-triggered sleep, bootstrap, refresh or retry; a normal accepted response remains the negative boundary.
- A reserved Redis task whose run reaches that terminal failure is ACKed exactly once without ready/processing residue or a recovery attempt. Detail challenges preserve already claimed candidates for a future new task, but never requeue or repeat the failed task itself.
- An isolated real PostgreSQL/Redis consumer scenario proves one task, one provider search, one failed run, invalid context payload and terminal ACK without Vinted, proxy or notification traffic.
- Completed runs are marked successful only when `execute_monitor_run()` returns `success`; failed runs do not reset proxy health.
- SQL persistence and Redis terminal/retry transitions form one recoverable state machine: any failure or crash leaves enough durable information for a later run to converge, without a successful event preceding an incomplete Redis transition.
- Baseline, session preparation and detail probes reject a durable `finalizing` run instead of starting a parallel non-reconciling flow. Task redelivery may resume only a persisted `worker_task_delivery_interrupted` attempt; classified response failures are already terminal and are acknowledged without new traffic.
- Prepared sessions are created by monitor launch/recalibration, are bound to monitor, proxy profile, proxy sticky id, browser profile, locale, viewport and Vinted `x-screen`, and are not reusable across monitors.
- Session invalidation immediately replaces encrypted cookies/tokens with an empty payload. Archiving and every session-context write take the same monitor row lock, so an in-flight refresh/preparation cannot recreate secrets after archive; all prepared sessions retain only safe lifecycle metadata.
- Source start/update/stop, scheduler enqueue and archive revalidate under the same PostgreSQL row lock. Stop/archive cancel a still-ready task best-effort; PostgreSQL archive and secret purge remain authoritative when Redis is unavailable.
- Automatic preparation attempts the DataDome collector only after the catalog document produced the base context needed for a conservative API request: CSRF, anon id, `access_token_web`, `v_udt`, `__cf_bm`, target/egress country match, locale, `Accept-Language`, viewport and `x-screen=catalog`. If any base context is missing, the collector is skipped and the session remains incomplete.
- The DataDome collector loads a returned `datadome` cookie into the same HTTP session before the prepared session is exported. If the collector returns no cookie, errors, or lacks the DataDome client key, the run continues to the catalog API probe only for diagnostics; the session remains `incomplete` because `datadome` is required.
- The catalog API probe reports `accepted_json`, `challenge`, `rejected`, `non_json`, or `transport_error` with safe request/response metadata. Only `accepted_json` plus complete prepared context marks the monitor-owned Vinted session ready.
- Business runs call the item HTML document detail path for every Redis-new candidate before creating an opportunity. Detail fetches use the prepared session cookie jar, sticky proxy identity, Chrome 146 document headers, `default_headers=False`, and the monitor catalog URL as referer.
- Detail navigation appends `referrer=catalog` without changing the canonical persisted item URL, rejects non-HTML success responses, and classifies `cf-mitigated: challenge` as a Cloudflare challenge rather than a normal HTTP error.
- Detail navigation validates the final response URL after redirects. Cross-origin, HTTP, credentialed, alternate-port, or non-item destinations are rejected before parsing.
- Response cookie rotation refreshes the in-memory context and the encrypted prepared-session row while retaining the reusable logical catalog screen.
- The item detail probe reports parser version, source map, required/optional missing fields, photo count, safe pricing fields, and availability state; it emits `detail_probe_finished` plus `run_succeeded` on success.
- Egress diagnostics never send the prepared Vinted cookie jar to the diagnostic endpoint. They may log the same proxy/session marker, response IP/country and safe response headers, but cookies from `access_token_web`, `refresh_token_web`, `datadome`, `anon_id`, `v_udt`, `__cf_bm` or the Vinted session jar must remain absent from the diagnostic request.
- A successful egress diagnostic is reusable for 300 seconds only by the same prepared session and proxy sticky id. Cookie refresh does not extend that validation; proxy/session failure, country mismatch or session invalidation clears it.
- Catalog API requests are blocked unless the prepared session has CSRF token, anon id, `access_token_web`, `v_udt`, `__cf_bm`, locale, egress country, `x-screen=catalog` and `datadome`. A DataDome challenge or rejection invalidates the session.
- Successful ordinary cookie rotations update the same encrypted `vinted_sessions` row with refreshed cookies/tokens/context, preserving the sticky proxy identity, request count and max request budget.
- Proxy failures use exponential cooldown; DataDome challenges use the configured challenge penalty multiplier.
- Run events expose safe diagnostics: `browser_profile`, `impersonate`, `vinted_session_id`, `vinted_session_use_count`, `session_end_reason`, `recovery_action`, `proxy_session_id_prefix`, target/proxy country, locale, `Accept-Language`, viewport, `vinted_screen`, response `x-screen`, DataDome/`__cf_bm`/CSRF/anon/access-token/v_udt presence booleans, probe outcome, rate-limit `Retry-After` fields, and masked/fingerprinted session markers only.
- Structured process logs apply the same redaction policy as persisted run events; exception strings never bypass the redactor.
- DataDome collector diagnostics expose only method, URL, status, duration, `jsType`, `ddv`, DataDome key presence/length, JS payload length, attempt count and cookie presence flags. `datadome_tags_request_*`, `datadome_collector_attempt_*`, and `datadome_collector_*` events never log raw cookies, tokens, `ddk`, `jspl`, collector response cookie values or full proxy credentials.
- Before calling `/api/v2/catalog/items`, the provider must have a complete prepared session context: Chrome `curl_cffi` impersonation, same-session egress diagnostic matching the target country, CSRF token, anon id, `access_token_web`, `v_udt`, `__cf_bm`, `datadome`, locale, `Accept-Language`, viewport and `vinted_screen=catalog`. Missing or contradictory base context records `catalog_session_context_incomplete` and blocks the catalog API request.
- Proxy create/update APIs reject legacy manual `locale`, `Accept-Language`, viewport and Vinted screen fields; changing `country_code` recomputes the stored read-only context.
- The PWA proxy test action is observable: clicking `Test IP` shows a pending state, disables conflicting proxy actions for that row, and leaves the latest success or failure visible without exposing credentials.
- The PWA monitor detail exposes a stopped-monitor `Probar detalle` control with an item id or URL input. It is blocked by unsaved monitor changes, unsupported URL filters or an active run, and leaves the latest detail probe result visible with accumulated logs.
- Before wiring a new ephemeral HTTP client into Redis workers or live Vinted catalog traffic, `scripts/verify_impersonation.py` exits successfully with exact Chrome 120 `User-Agent`, `sec-ch-ua`, and `Accept-Encoding` echoes, no Python/curl/cffi/requests leak markers in header values or non-browser header names, and expected BrowserLeaks TLS 1.3 / HTTP/2 fields. The standard Chrome header name `Upgrade-Insecure-Requests` is allowed.
- Manual and worker-owned catalog runs use the configured browser profile; the default is `chrome146`. Direct runs without a proxy are blocked by default and are available only when explicitly enabled by deployment configuration.
- Runtime browser profiles define ordered lowercase request headers for bootstrap and catalog API calls, and Vinted requests pass `default_headers=False` so `curl_cffi` owns TLS/HTTP2 impersonation but does not add duplicate browser headers. The code never manually sends HTTP/2 pseudo headers, `host`, `cookie`, `content-length`, or hop-by-hop headers such as `connection` or `te`; cookie values come only from the session cookie jar.

## Detail performance gate

- `serial` is the default detail execution mode even when a deployment configures concurrency greater than one.
- `canary` uses strict waves of at most two and validates the selected final cookie context with the catalog API before business commit. `parallel` is eligible only after controlled canaries reduce p50 batch duration by at least 25 percent without worse p95, challenge, `429`, parser mismatch, invalid session or Redis residue.
- A canary cookie/context incompatibility is recorded with safe fingerprints and leaves production in `serial`; it does not create permanent independent session lanes.
- The 2026-07-12 residential canary completed five details and final catalog validation with two lanes, but took `5.034 s` versus `4.598 s` for the real persistent-session C1 control. It therefore failed the performance promotion gate and production remains `serial`.
- Ten controlled early closes received only `15-22 KB` per rejected document in `455-636 ms` and preserved a final accepted catalog context. Enforcement still requires the separate 30-document semantic-equivalence gate; the default remains `shadow`.
- Detail timing separates HTTP, parser, filter and persistence. Events record counts, durations and safe fingerprints, never HTML, raw cookies, blacklist content or proxy credentials.
- Accepted opportunities read the public document to EOF. Early response termination is reserved for proven blacklist rejection because shipping, pricing, availability and the complete gallery occur near the end of observed Flight data.

## Known Limitations

- The deployment supports one worker instance with multiple in-process consumers. Independent worker replicas remain out of scope until queue reservations have distributed worker ownership and a visibility timeout.
- Stop/archive cancels work that is still in the ready list. A task already reserved immediately before stop can survive until its consumer acknowledges it; a future activation-generation field should distinguish the rare stop-edit-restart race before multi-instance deployment.
- REST, SSE and traffic-producing commands sit behind the local opaque PostgreSQL session boundary owned by `docs/specs/011-local-pwa-access-control.md`. This access session is independent of the anonymous Vinted context described here.
- Catalog/probe cookie rotation is not observed by the same refresh detector used for item documents, so PostgreSQL can retain stale context. Invalidation also lacks its own monitor-row lock. 14.12.4 closes every context mutation/write-order fence.
- Viewport is omitted from selection, repeated explicit preparation can leave several `ready` rows while runtime and PWA choose different ones, and the UI does not derive expiry/exhaustion or refresh automatic changes. 14.12.5 defines one canonical eligible read model.
- No durable key sentinel detects a global `APP_SECRET_KEY` mismatch before API/worker begin serving. A dependent read/operation must still stop with a clear safe error and no fallback; 14.12.6 is conditional early-startup detection for key rotation, long-lived data or server deployment.
- With a valid global key, invalid Vinted-session ciphertext/JSON can stay durably labelled `ready`, while one unreadable proxy password can fail summaries or egress construction. The affected operation must stop without retry loop or proxy penalty, and 14.12.5 must not present the row as usable. 14.12.7 is only the conditional durable corruption state; local repair may remove the affected row/profile manually.
- Proxy username is persisted and returned raw despite the credential contract. 14.12.8 minimizes and encrypts that read/write model.
- Reuse increments logical use before business traffic but commits it only with the later run transaction, so a crash or rollback can erase a consumed use. The personal local MVP accepts that rare visible drift and does not add the former 14.12.9 durable acquisition ledger.
- Preparation performs egress/bootstrap/probe before any attempt row or use exists, then records use one only for an accepted context. An abrupt crash may require manual relaunch and may repeat preparation; the former 14.12.10 exactly-once attempt owner is intentionally retired.
- Expired, exhausted and incomplete rows retain encrypted context until another explicit invalidation/archive action. 14.12.11 is conditional on meaningful row growth or a durable deployment rather than a prerequisite for current monitoring.
- Three obsolete proxy-owned session endpoints remain as `410` tombstones. Remove them opportunistically when those routes are next touched; they do not block current product work.
